Agent Surface is a practical guide to software built for agents: systems agents can understand, operate, evaluate, and safely extend.

The core idea is that agents do not interact with software through one neat interface. They read repository instructions, follow documentation, call APIs, run CLIs, inspect schemas, use MCP tools, parse errors, retrieve context, ask for approval, and coordinate across tools. Each contact point is a surface. If the surfaces are ambiguous, agents improvise. If the surfaces are explicit, agents can work reliably.

Agent Surface exists to make those surfaces intentional, and to connect them to the larger body of agent architecture, protocols, testing, tooling, and product design.

## The Story

Most software was built for human operators. A person can scan a docs page, infer missing context, recover from vague errors, and ask another person what a field means.

Agents are less forgiving. They need contracts, schemas, examples, recovery paths, permission boundaries, stable tool descriptions, and machine-readable context. They can be powerful when a system gives them clear handles. They become brittle when every step depends on guessing.

Agent Surface treats agent-readable software as an engineering discipline:

1. Name the surfaces agents touch.
2. Define what good looks like for each surface.
3. Choose the right runtime, protocol, tool, and data boundary.
4. Design recovery, approval, and evaluation paths up front.
5. Keep the surfaces current as products, models, and standards change.

### The `surface` Skill

`surface` is the operational part of Agent Surface. It can inspect a project, score agent-readiness, create a transformation plan, or scaffold agent infrastructure. Use it when you want the guide translated into concrete changes for a specific repository.

Scoring is useful, but it is not the whole site. The guide is broader: it explains the concepts, implementation patterns, standards, and evaluation methods behind agent-ready software.

## How to Use It

Use the documentation when you are learning agent architecture, designing an agent-facing surface, or comparing standards and tools.

Use the system design chapters when you are building agents.

Use the surface chapters when you are making an existing product usable by agents.

Use the scoring rubric when you need to evaluate a project or prioritize work.

Use `surface` when you want an evidence-backed audit, a concrete transformation plan, or to scaffold and update agent infrastructure in the shape of the existing project.

Use the templates when you are implementing a specific surface and want a starting point that is already shaped for agents.
