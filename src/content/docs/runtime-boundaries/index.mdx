## Summary

Agent architecture starts with a runtime boundary. The main question is not "which framework should run the agent?" It is "where should this work execute, how long may it run, what state may it touch, and who can observe or cancel it?"

- Run short, user-facing turns in request handlers only when they fit the platform timeout.
- Move long, retryable, or approval-gated work into background workflows.
- Keep durable state outside the model loop: database rows, workflow state, queues, or durable objects.
- Put browser and code execution in isolated runtimes with explicit budgets.
- Treat streaming, cancellation, auth, tracing, and idempotency as runtime requirements, not framework details.

## Boundary Types

| Runtime             | Best for                                                          | Avoid when                                               | Design pressure                                |
| ------------------- | ----------------------------------------------------------------- | -------------------------------------------------------- | ---------------------------------------------- |
| Request handler     | Short chat turns, routing, simple tool calls                      | Work may exceed timeout or needs retries                 | Low latency, clear auth, streaming             |
| Long-running server | Rich agents, streaming, many tools, local queues                  | Scale-to-zero or edge locality is required               | Process health, backpressure, deploys          |
| Serverless function | Stateless API tools, small LLM steps, webhooks                    | Stateful loops, large uploads, long workflows            | Cold starts, timeout budgets, retries          |
| Edge runtime        | Low-latency routing, lightweight model calls, content negotiation | Node-only packages, heavy compute, private network needs | Runtime limits, regional data policy           |
| Background worker   | Batch jobs, scheduled agents, retries, approvals                  | User is waiting for immediate answer                     | Idempotency, durable state, notifications      |
| Browser session     | Web navigation, screenshots, extraction, UI actions               | Data is available through an API                         | Permissions, isolation, audit logs             |
| Code sandbox        | Generated code, data transforms, notebooks, untrusted commands    | App server can safely run the code itself                | Secrets isolation, filesystem limits, teardown |

## Request-Bound Agents

Use a request handler when the interaction is short and user-visible:

- answer a question from already-available context
- call one or two bounded tools
- stream tokens or structured events to the UI
- classify intent and route to another system
- create a task record for slower work

Requirements:

- authenticate before constructing model context
- derive tenant, user, and resource IDs from trusted server state
- enforce a step budget
- support client cancellation
- return structured errors when model or tool calls fail
- emit spans for model calls, tools, retrieval, and final response

Do not hide multi-minute work inside a request. If the user needs progress, create a job and stream status from durable state.

## Background And Workflow Agents

Use a worker or durable workflow when the task needs time, retries, or checkpoints:

- document ingestion and indexing
- batch enrichment
- scheduled recommendations
- human approval before sending or purchasing
- long research tasks
- multi-step workflows that may pause and resume

The workflow should own:

- state transitions
- retry policy
- timeout policy
- idempotency keys
- human approval state
- final notification

Agents inside the workflow should own judgment, extraction, synthesis, or language generation. They should not be the only place where progress is tracked.

## Serverless Constraints

Serverless is a good fit for agent-accessible APIs and small LLM steps, but it is a poor fit for unbounded loops.

Watch for:

- hard execution timeouts
- cold starts that break streaming expectations
- missing local filesystem persistence
- retry semantics that can duplicate side effects
- regional limits around data residency
- provider-specific streaming behavior

When using serverless, make every side-effecting tool idempotent and write progress before doing irreversible work.

## Edge Constraints

Edge runtimes are useful for low-latency routing, discovery pages, Markdown content negotiation, auth checks, and lightweight model calls. They are risky for heavy agent work if the runtime does not support the packages, sockets, filesystem access, or execution time the agent needs.

Use edge when:

- latency to the user matters more than local compute
- the agent is mostly routing, transforming, or retrieving
- the data source is edge-local or globally replicated
- the platform supports the model provider and streaming shape you need

Avoid edge when:

- the agent needs Node-only SDKs
- tools require private network access
- work may run for minutes
- large files, browser automation, or code execution are involved

## Browser And Sandbox Boundaries

Browser and sandbox access should be separate runtime boundaries, not casual tools inside the main app server.

Minimum controls:

- allowlisted domains, commands, and network targets
- wall-clock and token budgets
- no production secrets in the session
- screenshots, command logs, and tool outputs retained for audit
- output truncation and artifact storage
- confirmation gates for writes, purchases, messages, and destructive actions
- deterministic teardown

Prefer APIs over browsers when APIs exist. Prefer deterministic transforms over generated code when the transform is known.

## Runtime Context

Trusted runtime context includes:

- user ID
- tenant or organization ID
- permissions and role
- resource IDs
- locale, timezone, and policy flags
- request ID and trace ID
- budget and model policy

This context should be assembled by server code and passed into tools or prompts deliberately. The model should not choose tenant IDs, memory keys, authorization scopes, or billing policy.

## Observability

Trace across runtime boundaries with a shared correlation ID:

- request or job start
- model calls
- tool calls
- retrieval queries
- workflow step transitions
- approval decisions
- retries and errors
- final output or notification

The trace should make it possible to answer: who asked, what context was used, which tools ran, what changed, what failed, and whether the output passed evaluation.

## Checklist

Before choosing a runtime, answer:

- How long can this work run?
- Can it be retried safely?
- What state must survive a crash?
- Does the user need streaming progress?
- Who can cancel it?
- Which secrets and resources can it access?
- What happens if a tool succeeds but the model call fails?
- Where are traces, artifacts, and eval results stored?
- Is this better as a deterministic workflow with agent steps?
