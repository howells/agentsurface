## Summary

MCP 2025-11-25 uses Client ID Metadata Documents (CIMD) where clients publish their identity and public key at `/.well-known/mcp-client.json`. Servers fetch the metadata, validate JWT signatures using the client's public key, and issue tokens without requiring shared secrets. Enterprise-Managed Authorization (EMA) centralizes this in an IdP for multi-tenant deployments.

```
Client publishes CIMD     Server discovers & verifies     Tokens issued
   at well-known          against public JWKS             with audit trail
```

- Client publishes CIMD: client_id, public_jwks_uri, requested_capabilities
- No shared secrets; clients authenticate via signed JWTs
- Short-lived access tokens (15 min–1 hour)
- EMA: optional, for enterprises with centralized IdP
- JWT-Bearer grant: client signs assertions with its private key

MCP's authorization model evolved from bearer tokens to dynamic registration to a standards-based metadata approach. MCP 2025-11-25 introduced Client ID Metadata Documents (CIMD) and Enterprise-Managed Authorization (EMA), solving the core problem: how do remote MCP servers securely authenticate clients without pre-shared secrets or complex dynamic registration?

The answer: clients publish metadata at a well-known location, servers discover and verify that metadata, tokens flow securely with full audit trails.

## Evolution of MCP Auth

**Early (2024):** Bearer tokens only. Clients passed a static token; servers accepted it without verifying client identity. High friction: every client-server pair needed out-of-band coordination.

**2025-06-18:** OAuth 2.0 dynamic client registration. Clients would register themselves at runtime, receive a `client_id` and `client_secret`, and use Client Credentials grant. Better, but required servers to expose a registration endpoint and trust clients to protect secrets.

**2025-11-25:** Client ID Metadata Documents + Enterprise-Managed Authorization. Clients publish their identity and capabilities at `/.well-known/mcp-client.json`. Servers fetch and verify this metadata, then issue tokens without requiring secrets to be shared. This is the current standard and recommended approach.

## Client ID Metadata Document (CIMD)

A client (the MCP server you're integrating with) publishes metadata at a well-known URL:

```json
{
  "name": "InvoiceSummaryAgent",
  "version": "1.0.0",
  "description": "Summarizes invoices for a given period",
  "client_id": "agent_01HV3K8MNP",
  "client_uri": "https://agents.example.com/invoice-summary",
  "public_jwks_uri": "https://agents.example.com/.well-known/jwks.json",
  "requested_capabilities": [
    "read:invoices",
    "read:customers"
  ],
  "redirect_uris": [],
  "contacts": [
    "admin@agents.example.com"
  ],
  "policy_uri": "https://agents.example.com/privacy",
  "tos_uri": "https://agents.example.com/tos"
}
```

Key fields:

- `client_id` — unique identifier for this client (UUIDv7 or ulid)
- `client_uri` — where the client is hosted or documented
- `public_jwks_uri` — where to fetch the client's public key for JWT verification
- `requested_capabilities` — which MCP server capabilities the client wants
- `redirect_uris` — empty for M2M (no browser redirect needed)

This metadata is static and serves as the client's "identity card." It is not secret and can be published in code repositories or documentation.

## Server-Side Metadata Fetch

When an MCP server receives a request from a client, it:

1. Extracts the client ID from the JWT
2. Fetches the client's metadata from `/.well-known/mcp-client.json` (or configured location)
3. Verifies the signature on the JWT using the public key from `public_jwks_uri`
4. Validates that the client is requesting only capabilities it declared
5. Issues appropriate permissions

```typescript
import { jwtVerify, createRemoteJWKSet } from 'jose';

async function authenticateClient(token: string): Promise<string> {
  // Decode token to get client_id (without verifying yet)
  const header = JSON.parse(
    Buffer.from(token.split('.')[0], 'base64').toString()
  );
  const payload = JSON.parse(
    Buffer.from(token.split('.')[1], 'base64').toString()
  );

  const clientId = payload.sub || payload.client_id;

  // Fetch client's metadata
  const metadata = await fetch(`https://client-domain.example.com/.well-known/mcp-client.json`)
    .then(r => r.json());

  if (metadata.client_id !== clientId) {
    throw new Error('Client metadata does not match token');
  }

  // Fetch client's public key
  const jwks = createRemoteJWKSet(new URL(metadata.public_jwks_uri));

  // Verify token signature
  const { payload: verified } = await jwtVerify(token, jwks, {
    audience: 'https://resource-server.example.com',
  });

  return verified.sub as string;
}
```

## Enterprise-Managed Authorization (EMA)

EMA is an optional extension where an organization's identity provider (IdP) manages authorization on behalf of all MCP servers and clients within that organization. Instead of individual servers fetching and verifying client metadata, the IdP issues tokens and servers trust the IdP.

In an EMA deployment:

1. Client authenticates to the organization's IdP
2. IdP issues a JWT with client identity and requested scopes
3. Client includes this JWT in requests to MCP servers
4. Servers validate the JWT against the IdP's JWKS
5. Servers check that the client is authorized for the requested operation

```typescript
// IdP issues a token
const token = await new SignJWT({
  sub: 'client_01HV3K8MNP',
  client_name: 'InvoiceSummaryAgent',
  aud: 'https://resource-server.example.com',
  scope: 'mcp:read mcp:invoke',
  org_id: 'org_acme',
  iat: Math.floor(Date.now() / 1000),
  exp: Math.floor(Date.now() / 1000) + 3600,
})
  .setProtectedHeader({ alg: 'RS256' })
  .sign(idpPrivateKey);

// Resource server validates
const { payload } = await jwtVerify(token, idpJwks, {
  issuer: 'https://idp.acme.org/',
  audience: 'https://resource-server.example.com',
});

// Resource server allows access based on org_id and scope
if (payload.org_id !== userOrgId) {
  throw new Error('Token org does not match resource org');
}
```

EMA is ideal for enterprises where multiple MCP servers operate within a single identity namespace and compliance requires centralized audit and authorization policies.

## Token Issuance with CIMD

When a client (MCP server) calls a resource server for the first time:

1. Client generates or loads a keypair
2. Client signs a JWT with its identity (using `sub` = client_id)
3. Client includes this JWT in the `Authorization` header
4. Resource server fetches the client's CIMD
5. Resource server validates the JWT signature using the client's public key
6. Resource server issues a short-lived access token

```typescript
// Client signs a JWT using its private key
const clientJwt = await new SignJWT({
  sub: 'agent_01HV3K8MNP',
  iss: 'https://agents.example.com',
  aud: 'https://resource-server.example.com',
  scope: 'read:invoices',
  iat: Math.floor(Date.now() / 1000),
  exp: Math.floor(Date.now() / 1000) + 60,
})
  .setProtectedHeader({ alg: 'ES256', kid: 'key_01HV3K8MNP' })
  .sign(clientPrivateKey);

// Client requests token
const tokenRes = await fetch('https://resource-server.example.com/oauth/token', {
  method: 'POST',
  headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
  body: new URLSearchParams({
    grant_type: 'urn:ietf:params:oauth:grant-type:jwt-bearer',
    assertion: clientJwt,
  }).toString(),
});

const { access_token } = await tokenRes.json();
```

The resource server validates the client JWT and issues a short-lived access token.

## MCP Server Endpoint for Authorization

An MCP server that implements CIMD should expose:

- `/.well-known/mcp-client.json` — the client's metadata (if the server is also a client calling other servers)
- `/.well-known/oauth-protected-resource` — if this server is a resource server
- `/oauth/token` — endpoint that exchanges client JWTs for access tokens (if this server is also an authorization server)

```typescript
// MCP server exposing its own metadata
server.setRequestHandler(
  {
    schema: {
      method: 'GET',
      path: '/.well-known/mcp-client.json',
    },
  },
  async () => {
    return {
      name: 'InvoiceSummaryAgent',
      version: '1.0.0',
      client_id: 'agent_01HV3K8MNP',
      client_uri: 'https://agents.example.com/invoice-summary',
      public_jwks_uri: 'https://agents.example.com/.well-known/jwks.json',
      requested_capabilities: ['read:invoices', 'read:customers'],
      contacts: ['admin@agents.example.com'],
    };
  }
);
```

## MCP 2025-11-25 Auth Specification

The MCP spec (https://spec.modelcontextprotocol.io/specification/2025-11-25/basic/authorization/) defines:

- **Client Metadata Discovery:** Clients publish CIMD at well-known locations
- **JWT-Bearer Grant:** Clients authenticate using signed JWTs (no shared secrets)
- **Scope and Capability Binding:** Requested scopes map to MCP server capabilities
- **Protected Resource Metadata:** Servers advertise auth requirements at `/.well-known/oauth-protected-resource`
- **Token Validation:** Servers validate JWTs and enforce scope checks per capability

MCP clients (like Claude Code's integrated servers) automatically:
- Load and expose their CIMD
- Sign JWTs for authentication
- Refresh access tokens before expiry
- Validate server metadata to determine auth requirements

## When to Use CIMD vs EMA

**Use CIMD when:**
- Clients and servers are independent organizations
- No centralized identity infrastructure exists
- You want clients to be self-describing and discoverable
- Tokens can be short-lived (15 min–1 hour)

**Use EMA when:**
- You operate a multi-tenant platform with a central IdP
- Compliance requires centralized audit and authorization
- You have many MCP servers under one identity namespace
- You need to revoke all client access centrally

## Checklist

- [ ] Client publishes CIMD at `/.well-known/mcp-client.json` (if acting as a client)
- [ ] CIMD includes `client_id`, `public_jwks_uri`, and `requested_capabilities`
- [ ] Client signs requests using JWT-Bearer grant (no shared secrets)
- [ ] Server fetches and validates client metadata before issuing tokens
- [ ] Server validates JWT signatures using the client's public key
- [ ] Server enforces that requested operations match declared capabilities
- [ ] Tokens are short-lived (15 min–1 hour)
- [ ] Server publishes `/.well-known/oauth-protected-resource` (if acting as a resource server)
- [ ] All auth interactions are logged with client_id and user_id for audit
