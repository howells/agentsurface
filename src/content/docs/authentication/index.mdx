## Summary

Agent authentication has two jobs: issue the right credential, then validate it on every request. Runtime API access should be programmatic: no login pages, CAPTCHAs, or session cookies in the request path. Registration can still include explicit human consent or an OTP claim ceremony when the goal is a user-bound credential. This section covers OAuth 2.1 Client Credentials for M2M, token exchange for delegation, DPoP for token binding, first-class agent identity for auditability, and emerging `auth.md` registration discovery.

- OAuth 2.1 Client Credentials with token caching for M2M auth
- Token Exchange (RFC 8693) for narrowing scope and delegation
- DPoP (RFC 9449) for sender-constrained tokens and replay protection
- Agent Identity: first-class principals, audit trails, and actor chains
- auth.md: agentic registration discovery for provider assertions and OTP claim flows
- Idempotency and replay protection: nonce stores and DPoP jti uniqueness

The key distinction is request path versus registration path. Agents should call APIs with bearer credentials, validated scopes, audience checks, and structured errors. If a user needs to approve or claim a delegation, keep that interaction bounded to registration and return a credential the agent can use programmatically afterward.

## Decision Tree

Start here: what kind of credential do you need?

- **I'm building a standalone agent service that calls APIs.** Use [OAuth 2.1 Client Credentials](/docs/authentication/oauth-for-agents). Short-lived tokens, narrow scopes, cached and refreshed proactively.

- **I have a simple third-party API and don't want OAuth complexity.** Use [API Keys](/docs/authentication/api-keys), but only if the key can be scoped, rotated, and short-lived.

- **An agent already has authority and needs to act on behalf of a user or delegate to a sub-agent.** Use [Token Exchange](/docs/authentication/token-exchange) (RFC 8693) to narrow scope and audience.

- **I need to prevent stolen tokens from being replayed.** Use [DPoP](/docs/authentication/dpop) (RFC 9449) to bind tokens to a client keypair.

- **My agents need to call multiple services and I need to audit who did what.** Use [Agent Identity](/docs/authentication/agent-identity) with distinct principals per agent.

- **I'm building an MCP server.** Publish [Protected Resource Metadata](/docs/authentication/protected-resource-metadata) at `/.well-known/oauth-protected-resource` and validate inbound JWTs.

- **I'm integrating with MCP's auth model.** See [MCP Authorization](/docs/authentication/mcp-auth-model) for OAuth protected-resource metadata, authorization-server discovery, and Client ID Metadata Documents.

- **An agent needs to register or claim a user-bound credential.** Publish [auth.md](/docs/authentication/auth-md), pair it with protected-resource metadata, and choose provider-attested or OTP-backed registration.
