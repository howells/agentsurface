---
title: Council Pattern
description: Running multiple agents independently on the same problem, then synthesizing or approving the final answer.
---

## Summary

A council asks several agents to solve or judge the same problem independently, then merges their outputs through a final synthesis, scoring, or approval step. Use it when answer quality matters more than latency and the problem benefits from distinct perspectives.

- Use councils for ambiguous, evaluative, or high-risk decisions.
- Keep every council member on the same input, but give each one a distinct lens.
- Require structured outputs so the synthesis step can compare results.
- Budget for duplicated work, higher token use, and slower responses.
- Prefer a workflow for the control plane; the LLMs should supply judgment, not hidden orchestration.

## When To Use It

Use a council when the system needs independent perspectives on the same question:

- code review across security, reliability, and maintainability
- policy review across legal, support, and product constraints
- medical, financial, or legal-adjacent triage where a human still owns the decision
- content quality review where one agent drafts and several agents judge the result
- model or prompt selection where candidates should be scored against the same rubric

Do not use a council for ordinary routing or predictable task decomposition. A supervisor is usually better when the problem should be split into different subtasks. A workflow is better when the path is known in advance.

## Control Flow

```
Input
  |-- Agent A: security lens ------|
  |-- Agent B: reliability lens ---|--> Synthesis or approval step --> Output
  |-- Agent C: product lens -------|
```

Each council member receives the same source input and returns a comparable result. The synthesis step should be explicit about whether it is merging, selecting, rejecting, or escalating.

## Implementation Shape

A council is easiest to reason about when it is implemented as a workflow. The workflow owns parallel execution, timeouts, result validation, and synthesis. The agents own only their domain judgment.

```typescript
import { z } from "zod";

const reviewSchema = z.object({
  verdict: z.enum(["pass", "revise", "block"]),
  confidence: z.number().min(0).max(1),
  findings: z.array(z.string()),
});

const synthesizeOutput = z.object({
  finalVerdict: z.enum(["pass", "revise", "block"]),
  requiredChanges: z.array(z.string()),
});

export async function runCouncilReview(change: string) {
  const [security, reliability, product] = await Promise.all([
    runAgent("security-reviewer", change, reviewSchema),
    runAgent("reliability-reviewer", change, reviewSchema),
    runAgent("product-reviewer", change, reviewSchema),
  ]);

  return runAgent("review-synthesizer", { security, reliability, product }, synthesizeOutput);
}
```

The exact framework can vary. In Mastra, this maps naturally to a workflow with `.parallel()` and a final synthesis step. In LangGraph, it maps to fan-out nodes with an accumulator and a synthesis node. In an app-local implementation, it can be plain typed functions as long as the boundaries are explicit.

## Synthesis Strategies

| Strategy       | Use when                                  | Risk                                         |
| -------------- | ----------------------------------------- | -------------------------------------------- |
| Majority vote  | The answer is categorical and low nuance  | Minority concerns can be real blockers       |
| Weighted score | Reviewers have known authority by domain  | Weights can hide weak evidence               |
| LLM synthesis  | Findings need to become prose or a plan   | The synthesizer can over-smooth disagreement |
| Human approval | The consequence of a wrong answer is high | Slower, but often the right product shape    |

When a council is used for review, default to preserving disagreement. A final answer that lists the blocking minority opinion is usually more useful than a polished consensus that erases risk.

## Failure Modes

- **Duplicated cost:** Councils intentionally repeat work. Set a maximum number of council members and a token budget.
- **Schema drift:** If agents return different shapes, the synthesis step becomes unreliable. Validate outputs before synthesis.
- **False consensus:** Agents can converge on the same plausible mistake. Use different prompts, models, or evidence sources when independence matters.
- **Over-synthesis:** The final agent may turn concrete findings into vague prose. Require it to preserve source findings and cite which council member raised each point.
- **No owner:** A council can expose tradeoffs without making a decision. Decide whether the synthesizer, a deterministic rule, or a human owns the final verdict.

## Design Checklist

- Every council member has a distinct lens.
- All members receive the same source input unless there is a clear reason not to.
- Outputs are structured and validated.
- The final step states how disagreement is handled.
- Budget and timeout behavior are explicit.
- High-impact decisions escalate to a human instead of relying only on synthesis.
