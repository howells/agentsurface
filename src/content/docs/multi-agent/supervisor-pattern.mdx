## Summary

A supervisor decomposes tasks, delegates to specialists via agents-as-tools or handoffs, collects results, and synthesizes output. Agents-as-tools (tool calls returning results) are standard for quick delegations; handoffs (transfer control, sub-agent talks to user) are for extended specialist sessions. Sub-agents are isolated from each other and receive only what the supervisor passes.

```
Supervisor (receives goal)
  ├─ Tool call: Research Agent → result 1
  ├─ Tool call: Analyst Agent → result 2
  └─ Tool call: Writer Agent → final output
```

- Agents-as-tools: supervisor receives output, can aggregate and reason
- Handoffs: supervisor transfers control, specialist owns conversation
- Memory isolation: sub-agents don't share state, pass data explicitly
- Over-delegation: avoid delegating every decision to sub-agents
- Validation: supervisor checks results meet requirements before proceeding

The supervisor pattern puts a single orchestrating agent in charge of decomposing a goal, delegating work to specialist sub-agents, collecting results, and synthesizing a final output. The supervisor is the only agent that communicates directly with the user or the calling system; workers are implementation details.

This is the most widely implemented multi-agent pattern across frameworks because it maps cleanly onto how teams work: a manager who understands the overall goal and specialists who execute specific tasks without needing to understand the full picture.

## Coordinator Control Loop

A production supervisor is more than a prompt that says "delegate to specialists." It needs a control loop that makes delegation observable, bounded, and testable.

```
User goal
  ↓
Supervisor chooses specialist
  ↓
Delegation start hook: shape task, add constraints, enforce policy
  ↓
Worker runs with filtered context
  ↓
Delegation complete hook: validate, log, retry, bail, or continue
  ↓
Completion scorer checks whether the overall task is done
  ↓
Supervisor synthesizes or delegates again
```

This pattern is useful for research coordinators, support triage, document production, incident review, and any workflow where one lead agent routes work across specialists.

## Worker Capability Descriptions

Sub-agent descriptions are routing contracts. The supervisor uses them to decide which worker should receive the next task, so they should describe both positive and negative scope.

Good worker descriptions include:

- what the worker does
- what it returns
- when to use it
- when not to use it
- expected output shape
- source or evidence requirements

Example:

```text
Research Specialist:
Use for gathering factual information, sources, statistics, and uncertainty notes.
Return concise bullet points with source URLs and confidence.
Do not write full narrative reports or make recommendations.

Writer:
Use after research exists.
Transform approved research into structured prose with headings and flow.
Do not invent facts, sources, or statistics.
```

Vague descriptions create routing drift. If every worker says "helps with research and writing," the supervisor has no reliable basis for delegation.

## Delegation Hooks

Delegation hooks are the policy boundary around sub-agent calls. They can be framework hooks, workflow steps, middleware, or plain functions, but the lifecycle is the same.

| Hook               | Purpose                                                       | Examples                                                                   |
| ------------------ | ------------------------------------------------------------- | -------------------------------------------------------------------------- |
| Before delegation  | Decide whether the call may proceed and shape the worker task | add recency constraints, remove unrelated context, enforce tenant policy   |
| After delegation   | Validate the result and decide next action                    | retry with feedback, bail on error, ask another worker, continue synthesis |
| Iteration complete | Monitor loop progress                                         | record finish reason, token use, step count, response length               |
| Task complete      | Decide whether the overall goal is satisfied                  | run scorers, verify required sections, check end state                     |

These hooks keep supervisor behavior from becoming an uninspectable model loop. They also give tests stable points to assert against.

## Context Filtering

Supervisors should not pass their entire transcript to every worker. Each delegation should receive the smallest context that lets the worker do its job.

Use context filters to:

- pass the current task and relevant prior results
- omit unrelated user messages
- remove hidden supervisor reasoning
- trim stale tool outputs
- redact credentials and private policy text
- summarize large artifacts before passing them forward

For research-to-writing flows, the writer usually needs approved findings, sources, style constraints, and target format. It does not need the supervisor's full routing history or failed research attempts unless those failures affect the final answer.

## Completion Scoring

Supervisors fail when they stop too early. Add a task-completion check that runs outside the supervisor's ordinary reasoning.

Completion scorers can be deterministic or model-graded:

| Scorer type      | Use for                                                                    |
| ---------------- | -------------------------------------------------------------------------- |
| Rule-based       | required sections, minimum citations, schema conformance, end-state checks |
| Retrieval-backed | claims supported by sources, no missing required documents                 |
| Model-graded     | open-ended completeness, clarity, relevance, risk review                   |
| Human review     | high-impact or externally visible output                                   |

For a research coordinator, completion criteria might include:

- topic scope is fully addressed
- claims cite sources
- uncertainties are named
- final output has the requested structure
- no unsupported statistics appear
- writing step used the approved research rather than inventing new facts

If the scorer says the task is incomplete, the supervisor should receive targeted feedback and continue with a bounded retry. If the same failure repeats, stop and return a structured error or ask for human input.

## The Core Distinction: Handoffs vs. Agents-as-Tools

Before choosing a framework or runtime, the most important conceptual distinction in supervisor-pattern work is how control moves:

**Handoffs (transfer control):** The supervisor transfers execution to a sub-agent and stops. The sub-agent communicates directly with the user until it transfers control back or concludes. The supervisor does not receive the sub-agent's output programmatically — it re-enters when the sub-agent initiates a return handoff.

**Agents-as-tools (return control):** The supervisor invokes a sub-agent as a tool call. The sub-agent runs to completion and returns a result. The supervisor receives that result, adds it to its context, and continues reasoning. The sub-agent never communicates directly with the user.

|                    | Handoff                                 | Agent-as-Tool                             |
| ------------------ | --------------------------------------- | ----------------------------------------- |
| Who talks to user  | Sub-agent after transfer                | Supervisor only                           |
| Result flow        | Back via return handoff                 | Returned as tool output                   |
| Context continuity | Sub-agent has full conversation history | Sub-agent has only what supervisor passes |
| Use case           | Deep specialist sessions                | Quick delegated lookups                   |

Most production supervisor implementations use agents-as-tools for the orchestration layer, reserving handoffs for cases where the specialist needs extended back-and-forth with the user.

## Framework Mapping

Different frameworks expose the supervisor pattern with different nouns, but the design questions are the same: who owns control, how workers are described, what context they receive, how results return, and where completion is checked.

| Framework family      | Common vocabulary                                  | Guide-level concern                                                                                        |
| --------------------- | -------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| Mastra-style agents   | supervisor agent, sub-agents, delegation hooks     | Describe worker capabilities clearly, observe delegation lifecycle, and keep memory boundaries explicit.   |
| AI SDK-style apps     | agents as tools, tool output transformation        | Treat worker calls as typed tools and pass only decision-relevant summaries back to the supervisor.        |
| Graph runtimes        | supervisor node, worker nodes, conditional routing | Keep state transitions inspectable and make termination a graph decision, not only a prose decision.       |
| Crew-style frameworks | manager, crew, tasks, expected outputs             | Make task ownership and expected outputs explicit enough that the manager can route and verify work.       |
| Handoff-first SDKs    | handoffs, agents as tools                          | Choose transfer-of-control for extended specialist sessions and return-control for bounded delegated work. |

Use framework docs for exact APIs. In Agent Surface, the durable reference is the control contract: worker descriptions, context filters, hooks, validation, completion scoring, and observability.

## Failure Modes Common to All Implementations

**Over-delegation:** The supervisor delegates every decision to sub-agents, including decisions it could make directly from available context. Each delegation adds latency, cost, and an opportunity for failure. Supervisors should delegate tasks, not questions.

**Under-specified sub-agents:** A sub-agent whose instructions do not clearly scope its role will attempt to answer questions outside its domain, producing unreliable results. Every sub-agent's instructions should define both what it does and what it does not do.

**Context leakage across delegations:** When a supervisor accumulates tool results from multiple sub-agents in its context, earlier results affect later delegations through the supervisor's reasoning. If the research agent returns uncertain findings, this uncertainty can incorrectly bias the analyst's instructions. Structure delegation prompts to isolate each sub-agent's inputs.

**No result validation:** The supervisor receives a sub-agent's output and passes it directly to the next stage without checking whether it met the task requirements. Add explicit validation at the supervisor level: "Did the researcher return the confidence level I asked for? If not, retry with a clarified request."

**No completion gate:** The supervisor decides the task is done because it has produced fluent prose, not because the output satisfies the user's goal. Add task-completion scoring or a deterministic checklist before final response.

**Unbounded context inheritance:** Each worker receives the full conversation and all prior worker outputs. This increases cost and causes workers to react to irrelevant context. Filter delegation context per worker.
